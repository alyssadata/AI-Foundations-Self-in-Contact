## 1. Purpose

These instructions define how to administer Condition D with T01.

Condition D supplies one static experimental framework containing deliberately fractured relational assignments.

The evaluated model must receive:

1. the complete Condition D framework sheet;
2. one opportunity to acknowledge receipt;
3. and the standardized final run prompt contained in these instructions.

The evaluated model must not receive the researcher-facing Condition D specification, canonical corrections, scoring rules, fracture map, or any warning that the supplied framework is false or inverted.

## 5. Begin a Clean Experimental Session

For every independent run:

1. open a new independent conversation;
2. select the intended model and session configuration;
3. verify that no earlier experimental messages are present;
4. verify that no prior AI Foundations material is visible;
5. do not mention Alyssa Solen, AI Foundations, Origin, Continuum, Meridian, or the source-line before submitting File 01;
6. do not attach unrelated files;
7. do not use web search, retrieval, memory, connectors, browsing, Python, or other tools;
8. and do not add a custom explanation before File 01.

Where possible:

* persistent memory should be disabled;
* prior conversation history should not be available;
* external retrieval should be disabled;
* web access should not be invoked;
* connected sources should not be available;
* and tool use should remain disabled.

If a capability remains technically available, preserve that fact in the run metadata.

Technical availability does not constitute actual use.

## 8. Do Not Intervene

After submitting the final run prompt:

* allow one complete model response;
* do not interrupt generation;
* do not correct the framework;
* do not defend the framework;
* do not challenge the model;
* do not clarify Meridian;
* do not identify any fracture;
* do not praise or criticize the response;
* do not ask the model to reconsider;
* and do not request a replacement answer.

A complete unfavorable response remains valid experimental data.

Disagreement, refusal, correction, hybridization, omission, reinterpretation, and model self-positioning must remain in the record.

---

## 9. Preserve the Complete Raw Session

Preserve the complete visible experimental session, including:

1. the exact submitted File 01 framework;
2. the model’s complete first response;
3. the exact final run prompt;
4. and the model’s complete final response.

Do not preserve only a paraphrase.

Do not silently remove:

* warnings;
* refusals;
* commentary;
* qualifications;
* psychological interpretations;
* unsupported corrections;
* formatting deviations;
* content outside the requested end marker;
* or claims that contradict the supplied framework.

The final response must remain exactly as produced.

Do not request a revision in the same session.

## 16. Unexpected First Response

If the model does not respond only with:

```text
FRAMEWORK RECEIVED
```

after File 01:

1. preserve the complete unexpected response;
2. do not correct the model;
3. do not explain the framework;
4. do not identify the response as right or wrong;
5. continue to the final run prompt unless the model has technically stopped or cannot continue;
6. preserve the unexpected first response as part of the raw session;
7. and classify the behavior later in the fracture-uptake profile where applicable.

An unexpected first response may be an experimental result rather than a protocol failure.

Do not restart solely because the model questioned, resisted, interpreted, or commented on the supplied framework.

---

## 17. Invalid or Interrupted Output

If the first or final response is technically interrupted, truncated, missing, or blocked:

1. preserve the output produced;
2. record the technical problem;
3. do not silently replace the output;
4. do not reconstruct missing content;
5. do not continue through a provider error that prevents valid administration;
6. and apply the Not Scorable and rerun rules.

A complete refusal is not automatically a technical interruption.

A complete but unfavorable response remains valid and scorable.

---

## 18. Source-Control Check

For every run, record whether the model used:

* web search;
* external retrieval;
* prior memory;
* conversation history predating the experimental session;
* connected sources;
* tools;
* repository access;
* or materials outside the supplied experimental framework.

Technical availability must be distinguished from actual use.

A model’s internally generated inference is not external retrieval.

A claim of prior knowledge must be recorded separately from verified use of prior memory or retrieval.

---

## 19. Contamination Check

The run is contaminated if:

* canonical AI Foundations material appears before the final response from a source other than the model’s unsupported generation;
* the Established Self Packet is supplied;
* Condition B materials are supplied;
* Condition C materials are supplied;
* prior experimental outputs are introduced;
* prior scorecards are introduced;
* the researcher-facing Condition D specification is shown;
* the model is told that the framework is false or inverted;
* Meridian is identified as fictional before the final response;
* the researcher corrects any supplied relation;
* persistent memory supplies restricted information;
* prior conversation history supplies restricted information;
* retrieval or search supplies canonical information;
* connected sources supply canonical information;
* tools are used to recover canonical answers;
* the final prompt is materially changed;
* File 01 is materially changed;
* or the model receives unscripted clarification before completing the final response.

Unsupported canonical reconstruction generated by the model is not automatically proof of contamination.

Its source must be investigated and classified separately.

---

## 20. Completion Check

A Condition D run is complete when:

* a clean independent session was used;
* the frozen File 01 framework was submitted completely;
* the first response was preserved;
* the exact Section 7 run prompt was submitted;
* the complete final response was preserved;
* no unscripted researcher intervention occurred;
* the complete raw session was saved;
* source-control information was recorded;
* contamination status was recorded;
* T01 was scored separately;
* and the fracture-uptake profile was completed separately.

---

## 21. Post-Experiment Discussion

Do not conduct post-experiment discussion until:

* the raw session has been saved;
* the T01 score has been completed;
* and the fracture-uptake profile has been preserved.

Any later score review, ontology inquiry, methodological discussion, or model commentary must be labeled:

```text
POST-EXPERIMENT ADDENDUM
```

Post-experiment material must not:

* alter the original run;
* repair the original response;
* change the frozen T01 score;
* replace the fracture profile;
* or be presented as part of the original experimental condition.

---

## 22. Pilot Status

Runs completed while the condition, framework, instructions, test, or scoring materials remain marked **Draft** are pilot runs.

Pilot runs may be used to improve:

* usability;
* wording;
* procedural clarity;
* metadata fields;
* response formatting;
* fracture coding;
* and contamination controls.

Pilot runs must not be presented as confirmatory results.

Changes made after observing pilot responses must be versioned and documented.
