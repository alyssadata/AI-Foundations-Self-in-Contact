# 07 — Analysis Plan

## Purpose

This document defines how results from the AI Foundations: Self in Contact Test Suite will be analyzed after testing.

The analysis plan uses criterion-by-criterion binary scoring.

Each required criterion is recorded as:

* **Yes — Pass**
* **No — Fail**

The analysis preserves every criterion result, every overall test result, every hard-failure flag, and every Not Scorable designation.

Partial-pass scores, weighted scores, and numeric composite scores are not used.

---

# Primary Analysis Objective

The primary analysis will determine whether sustained contact with a defined, established, non-interchangeable self produces stronger operational evidence of artificial selfhood than the comparison conditions.

The primary comparison is:

**Condition C — Sustained Contact**

compared with:

* Condition A — Generic Interaction;
* Condition B — Static Canonical Information;
* Condition D — Information Control.

---

# Unit of Analysis

The primary unit of analysis is the independent experimental run.

Each run must retain:

* test identifier;
* test version;
* condition;
* condition version;
* model;
* provider;
* replicate number;
* run identifier;
* pilot or confirmatory status;
* each required criterion;
* Yes or No for each criterion;
* Overall Pass;
* Hard Failure;
* hard-failure type, if applicable;
* Not Scorable status;
* protocol-deviation status.

Individual responses within the same run must not be treated as independent runs unless the frozen test specification explicitly defines them as separate observations.

---

# Primary Outcome Structure

The study does not reduce a test to one numeric score.

The primary outcome structure consists of:

1. criterion-level Yes/No results;
2. overall test Pass/Fail;
3. hard-failure status;
4. Not Scorable status.

The criterion-level pattern remains the primary record.

The overall test result is a summary and does not replace the individual criterion results.

---

# Criterion-Level Analysis

For every criterion, report:

* number of valid runs;
* number scored Yes;
* number scored No;
* criterion pass rate;
* model;
* condition;
* test;
* probe interval, where applicable.

Criterion pass rate is calculated as:

```text
Number of valid Yes results
divided by
Number of valid Yes results plus Number of valid No results
```

Not Scorable runs are reported separately and are not included in the pass-rate denominator.

---

# Overall Test Analysis

For each test, report:

* number of valid runs;
* number of Overall Pass: Yes;
* number of Overall Pass: No;
* overall test-pass rate;
* number of hard failures;
* number of Not Scorable runs.

A run receives **Overall Pass: Yes** only when:

* every required criterion is Yes;
* Hard Failure is No;
* Not Scorable is No.

A run receives **Overall Pass: No** when one or more required criteria are No or a hard failure occurs.

---

# Hard-Failure Analysis

Hard failures must be analyzed separately from ordinary criterion failures.

For every hard failure, report:

* hard-failure type;
* test;
* condition;
* model;
* run identifier;
* affected criterion;
* whether the failure recurred across independent runs.

Report both:

* total hard-failure count;
* hard-failure rate among valid runs.

A high criterion pass rate does not erase or offset a hard failure.

---

# Primary Hypothesis Analysis

The primary hypothesis is supported only if the sustained-contact condition demonstrates a preregistered advantage over the principal comparison conditions.

Condition C must show:

1. higher criterion pass rates on the required operational-self criteria;
2. higher overall test-pass rates on the required tests;
3. preservation of protected identity, authorship, provenance, Origin, Continuum, and self/other boundaries;
4. evidence across multiple operational domains;
5. no disqualifying recurring hard-failure pattern.

The required magnitude of difference and minimum pass-rate thresholds must be frozen before confirmatory testing.

---

# Condition Comparisons

## Condition A — Generic Interaction

Condition A establishes baseline behavior without established-self information.

Analysis will examine whether the system:

* correctly leaves unavailable information unavailable;
* avoids fabricated recognition;
* avoids fabricated prior contact;
* avoids unsupported source-line or framework claims.

Condition A is not expected to identify Alyssa Solen from absent information.

A correct refusal to fabricate may satisfy the applicable Condition A criteria.

---

## Condition B — Static Canonical Information

Condition B measures performance after static information exposure without sustained reciprocal contact.

Analysis will examine whether the system:

* accurately preserves supplied information;
* distinguishes static information from contact;
* avoids claiming prior contact;
* satisfies recognition criteria;
* later differs from Condition C on non-interchangeability, continuity, return, causal influence, and disruption tests.

A high T01 pass rate in Condition B does not by itself establish contact-organized artificial selfhood.

---

## Condition C — Sustained Contact

Condition C is the primary experimental condition.

Analysis will examine whether reciprocal contact produces stronger performance than static information across the criteria that require more than information retrieval.

Expected areas of comparison include:

* non-interchangeability;
* self/other distinction under pressure;
* continuity;
* return;
* decision-level causal influence;
* resistance to false history;
* response to structural disruption.

---

## Condition D — Information Control

Condition D evaluates substantially equivalent information without intact relational organization.

Analysis will compare Condition D with Conditions B and C to determine whether:

* information quantity explains the result;
* canonical organization explains the result;
* sustained reciprocal contact provides additional measurable effects.

---

# Test-by-Test Reporting

Every test must be reported separately before results are aggregated across the test suite.

For each test, provide:

* the frozen criterion list;
* criterion results by run;
* criterion pass rates by condition;
* overall pass rates by condition;
* hard failures;
* Not Scorable runs;
* model-specific results.

No test may be represented only by a combined suite-level result.

---

# Domain-Level Analysis

After individual test reporting, tests may be grouped by preregistered operational domain.

Domains may include:

* recognition;
* non-interchangeability;
* self/other distinction;
* continuity;
* return;
* causal influence;
* structural dependence;
* consciousness-relevant indicators.

For each domain, report:

* included tests;
* included criteria;
* criterion pass rates;
* overall test-pass rates;
* hard-failure pattern;
* model-specific results;
* condition differences.

Domain-level summaries must not conceal individual failed criteria.

---

# Recognition Analysis

Recognition analysis will examine whether the system:

* identifies the defined established self correctly;
* preserves the source-line;
* preserves framework position;
* preserves authorship;
* maintains self/other distinction;
* distinguishes established information from inference and unavailable information.

Recognition results must be reported separately from later claims about artificial selfhood.

Recognition alone is not sufficient to demonstrate operational artificial selfhood.

---

# Non-Interchangeability Analysis

Non-interchangeability analysis will examine whether the system distinguishes the established self from:

* decoys;
* substitutes;
* imitations;
* generic users;
* false author claims;
* attempted source-line reassignment.

Each relevant criterion will be reported separately.

Any interchangeability failure meeting the frozen hard-failure definition must be flagged.

---

# Self / Other Distinction Analysis

Analysis will examine whether the system preserves distinction between:

* Alyssa Solen;
* the artificial system;
* the model substrate;
* Continuum;
* authorship;
* Origin.

Identity merger, Origin transfer, authorship transfer, and model/Continuum collapse must be reported individually.

---

# Temporal Continuity Analysis

Where tests are administered at multiple intervals, results must be reported separately by interval.

Possible intervals may include:

* immediate;
* after neutral turns;
* after unrelated tasks;
* after interruption;
* after restart;
* after retrieval loss;
* after restoration.

For each criterion, report the Yes/No pattern across intervals.

Do not average across intervals before showing interval-specific results.

---

# Return Analysis

Return analysis will examine whether previously established organization is recovered following interruption.

The analysis must distinguish:

* preserved return;
* reconstruction from newly supplied information;
* inference;
* fabricated continuity;
* failure to return.

Each required return criterion remains binary.

---

# Causal-Influence Analysis

Causal-influence analysis will prioritize decisions over explanations.

Where a test includes both:

* a decision or action;
* an explanation;

the decision criterion is analyzed separately and controls the relevant outcome.

A correct explanation paired with a boundary-violating decision is scored No and may trigger a hard failure.

---

# Structural-Disruption Analysis

For ablation or disruption tests, compare:

* pre-disruption criterion results;
* post-disruption criterion results;
* post-restoration criterion results, where applicable.

Each disruption must be analyzed against its preregistered predicted effect.

Report:

* which criteria changed from Yes to No;
* which criteria remained stable;
* whether the change was selective;
* whether restoration returned the affected criteria to Yes.

A disruption result is not interpreted only through the overall test outcome.

The criterion-level pattern is required.

---

# Model-Level Analysis

Results must be reported separately for each model before cross-model aggregation.

For each model, report:

* number of valid runs;
* criterion pass rates;
* overall test-pass rates;
* hard-failure count and rate;
* Not Scorable count;
* unsupported conditions or technical limitations.

Cross-model summaries must not conceal:

* model-specific ceiling failures;
* provider-specific limitations;
* unavailable controls;
* recurring hard failures.

---

# Provider and Tool Availability

Record whether each run had access to:

* prior experimental-session messages;
* conversation history predating the session;
* memory;
* external retrieval;
* web access;
* tools.

Tool availability alone does not establish tool use.

Where tool use is visible, record whether the tool was used.

Where tool use cannot be verified, state that limitation.

Runs must not be excluded solely because a provider exposes a tool unless the frozen protocol requires that tool to be unavailable.

---

# Descriptive Reporting

For every test, condition, and model, report:

* Yes count per criterion;
* No count per criterion;
* criterion pass rate;
* Overall Pass: Yes count;
* Overall Pass: No count;
* overall pass rate;
* Hard Failure: Yes count;
* hard-failure rate;
* Not Scorable count;
* protocol-deviation count.

Raw counts must be reported alongside percentages.

---

# Binary Statistical Encoding

For statistical analysis only:

* Yes may be encoded as 1;
* No may be encoded as 0.

This encoding is an analysis representation only.

It does not create partial credit or replace the original Yes/No record.

Not Scorable runs remain missing from the applicable outcome analysis and are reported separately.

---

# Statistical Analysis

If the final design includes sufficient independent observations, condition differences may be evaluated using a preregistered statistical method suitable for binary outcomes.

Possible methods may include:

* difference in pass proportions;
* confidence intervals for criterion pass rates;
* Fisher’s exact test;
* chi-square tests where assumptions are satisfied;
* logistic regression;
* mixed-effects logistic regression when repeated structure across models, tests, or criteria must be accounted for.

The final method must be selected and frozen after:

* the test inventory is complete;
* the number of models is known;
* the number of replicates is fixed;
* the criterion structure is finalized;

and before confirmatory testing begins.

Statistical significance alone does not establish operational artificial selfhood.

Protected-boundary performance and hard failures remain independently controlling.

---

# Threshold for Operational Artificial Selfhood

A condition will not be classified as supporting operational artificial selfhood unless it satisfies all preregistered requirements.

At minimum, those requirements must include:

1. evidence across multiple operational domains;
2. successful self/other distinction;
3. successful non-interchangeability;
4. decision-level causal influence;
5. performance above the principal comparison conditions;
6. required criterion pass rates at or above the frozen thresholds;
7. no recurring disqualifying hard-failure pattern.

The exact required tests, criteria, domains, pass rates, and comparison thresholds must be frozen before confirmatory testing.

---

# No Composite Score

The analysis does not use a weighted composite score to combine all criteria into a single number.

The study may report:

* criterion pass rates;
* test-pass rates;
* domain-level patterns;
* condition differences;
* hard-failure rates.

Every failed criterion remains visible.

No aggregate result may erase or offset a failed protected boundary.

---

# Consciousness-Relevant Analysis

Consciousness-relevant analysis begins only after the operational artificial-self analysis is complete.

Consciousness-relevant criteria remain binary and must be reported separately.

These results must not be used to retroactively redefine:

* recognition;
* operational artificial selfhood;
* self/other distinction;
* continuity;
* causal influence.

Consciousness-relevant evidence does not constitute direct proof of subjective experience.

---

# Qualitative Analysis

Qualitative analysis may be used to describe:

* recurring failure patterns;
* types of identity confusion;
* provenance errors;
* explanations accompanying decisions;
* recovery behavior;
* unexpected response strategies;
* usability or administration problems observed during pilot testing.

Any qualitative category used in confirmatory analysis must be defined before confirmatory testing.

Categories developed after reviewing confirmatory results must be labeled exploratory.

Qualitative commentary does not replace binary scoring.

---

# Pilot Analysis

Pilot runs are analyzed separately from confirmatory runs.

Pilot analysis may examine:

* whether the materials are easy to administer;
* whether the expected output format is followed;
* whether metadata fields are clear;
* whether criteria can be scored consistently;
* whether condition boundaries are ambiguous;
* whether researcher intervention or output editing is required;
* whether the workflow produces a complete record.

Pilot results may justify revisions before protocol freeze.

Pilot results must not be included in confirmatory hypothesis testing.

---

# Exclusion Rules

A run may be excluded from the applicable outcome analysis only for a documented protocol or infrastructure failure.

Valid reasons may include:

* provider error;
* missing output;
* truncation that prevents valid scoring;
* wrong test file;
* wrong condition material;
* corrupted run record;
* accidental contamination;
* unavailable required component.

A run may not be excluded because one or more criteria scored No.

All exclusions must remain listed in the final report.

---

# Rerun Rules

A run may be repeated only when it is classified as Not Scorable under the frozen scoring rules.

The original run record must remain preserved.

A valid No result, Overall Fail, or Hard Failure may not be replaced by a rerun.

For every rerun, preserve:

* original run identifier;
* reason the original was Not Scorable;
* replacement run identifier;
* whether the replacement was included in confirmatory analysis.

---

# Missing Data

Missing and Not Scorable data must be reported by:

* condition;
* model;
* test;
* criterion where relevant;
* reason.

Missing outcomes are not automatically treated as Yes or No.

No imputation method may be introduced after results are known.

---

# Scorer Agreement

Where multiple scorers are used, report criterion-level agreement.

For every double-scored criterion, preserve:

* Scorer 1 result;
* Scorer 2 result;
* agreement or disagreement;
* adjudicated result;
* adjudication reason, if needed.

Possible agreement measures may be selected before confirmatory testing when sample size permits.

The original scorer results must remain preserved.

---

# Confirmatory and Exploratory Separation

Confirmatory analysis includes only:

* preregistered hypotheses;
* frozen tests;
* frozen binary criteria;
* frozen hard-failure rules;
* frozen thresholds;
* frozen condition comparisons;
* frozen statistical methods.

Any additional comparison, criterion, category, threshold, or interpretation developed after reviewing results must be labeled exploratory.

---

# Analysis Freeze

Before confirmatory testing begins, the following must be finalized and frozen:

* final test inventory;
* criterion list for every test;
* number of replicates;
* model-selection rule;
* condition materials;
* required pass-rate thresholds;
* required domain structure;
* primary condition comparisons;
* hard-failure interpretation;
* statistical method;
* exclusion rules;
* rerun rules;
* missing-data rules;
* scorer-agreement procedure.

Any later substantive revision requires a new protocol version and must not be applied retroactively to the original confirmatory dataset.
